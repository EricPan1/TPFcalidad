1. Propósito y Contenido del Documento

1.1 Propósito

El presente Plan de Pruebas define la estrategia, actividades, recursos, calendario y entregables del proceso de control de calidad para la aplicación de Gestión de Personas desarrollada sobre la plataforma OrangeHRM, extendida con un widget de información climática provisto por la API WeatherAPI.com.

Su propósito específico es:

  • Describir qué se probará, cómo, cuándo y con qué herramientas en cada sprint.
  • Definir los criterios de entrada, salida, suspensión y reanudación de las actividades de prueba.
  • Establecer la distribución de responsabilidades del equipo de QA.
  • Proveer una guía de referencia que permita a stakeholders y profesores revisar y comprender el proceso de testing aplicado.

1.2 Contenido del documento

2. Alcance

2.1 Dentro del alcance (in-scope)

  • Pruebas funcionales de frontend (E2E automatizadas con Playwright + Chromium) para los módulos PIM y Admin de OrangeHRM.
  • Pruebas de contrato de API (Newman / Postman Collection v2.1) para los cuatro endpoints de WeatherAPI.com.
  • Pruebas exploratorias manuales en paralelo al desarrollo durante cada sprint.
  • Pruebas de regresión acumulada: en cada sprint se re-ejecuta la suite del sprint anterior.
  • Smoke testing: subconjunto crítico de casos que corre ante cada merge a rama principal.
  • Gestión y reporte de defectos mediante el Bug Tracker integrado.

2.2 Fuera del alcance (out-of-scope)

  • Pruebas de performance y carga (cubiertas en las respuestas a consignas 4 y 5 del TP).
  • Pruebas de seguridad exhaustivas (OWASP Top 10). Se realiza una verificación básica de validación de roles en Sprint N+2.
  • Pruebas de frontend mobile (OrangeHRM presenta funcionalidades acotadas en mobile; quedan para un plan complementario).
  • Pruebas de accesibilidad (WCAG).

3. Objeto de Prueba

3.1 Descripción de la aplicación

La aplicación de Gestión de Personas opera a través de microservicios que proveen funcionalidades específicas. Expone a los usuarios:

  • Frontend web (desktop): interfaz principal de administración y autogestión.
  • Frontend web mobile: versión con funcionalidades acotadas para dispositivos móviles.
  • Backend del frontend: servicios REST que alimentan ambos frontends.
  • Widget de clima (Sprint N+3): integración con WeatherAPI.com desde el dashboard principal.

4. Estrategia de Pruebas

4.1 Pirámide de testing

La estrategia sigue el modelo de pirámide de pruebas: mayor cantidad de pruebas en niveles más bajos (rápidas y baratas) y menor cantidad en los niveles superiores (lentas y costosas).

E2E / UI · Playwright (25 casos)
API / Contrato · Newman (20 casos)
Exploratoria Manual (por sprint)
Unit / Integración · Responsabilidad del equipo de Desarrollo

4.3 Enfoque Shift-Left

¿Qué significa? El testing no es una fase final, sino una actividad permanente que comienza el día 1 del sprint. Los testers participan desde el refinamiento, revisando criterios de aceptación, para detectar ambigüedades antes de que se codifiquen. Las pruebas tempranas contra mocks del backend permiten validar el frontend antes de que el backend esté listo.

4.6 Datos de prueba

  • Los tests que crean empleados usan el prefijo AutoTest_ + timestamp (ej: AutoTest_1717600000) para ser identificables y evitar colisiones con otros testers del demo.
  • Las búsquedas usan el primer registro existente de la tabla (no dependen de datos fijos).
  • Los tests de API usan la ciudad Buenos Aires como caso válido y una cadena ficticia (XYZ_CIUDAD_INEXISTENTE_12345) para casos inválidos.
  • La fecha histórica fija 2024-01-15 se usa para el endpoint /history.json.

5. Criterios de Entrada y Salida

5.1 Criterios de entrada (por sprint)

Para iniciar la ejecución formal de pruebas en un sprint se debe cumplir:

  • Las historias de usuario del sprint tienen criterios de aceptación documentados y revisados por QA.
  • El ambiente de testing está disponible y respondiendo (smoke básico de infraestructura).
  • Los casos de prueba del sprint están diseñados y revisados (al menos el 80%).
  • Las historias marcadas como "listas para probar" han pasado code review.
  • Los mocks del backend están disponibles para la primera fase de pruebas tempranas.

5.2 Criterios de salida (por sprint)

El sprint se considera listo para cerrar desde la perspectiva de QA cuando:

CriterioUmbral mínimo
Casos de smoke ejecutados y pasando100%
Casos de regresión (sprints anteriores) ejecutados y pasando100%
Bugs CRITICAL o HIGH abiertos0
Bugs MEDIUM abiertos sin workaround documentado0
Reporte de ejecución disponible en AllureSí
Bugs encontrados registrados en Bug Tracker con pasos reproduciblesSí

5.3 Criterios de suspensión

  • El ambiente de pruebas no está disponible por más de 4 horas consecutivas.
  • Se detecta un bug CRITICAL que bloquea la ejecución de la mayoría de los casos.
  • Más del 30% de los casos de smoke fallan por razones no relacionadas con los cambios del sprint.

5.4 Criterios de reanudación

  • El ambiente es restaurado y verificado.
  • El bug CRITICAL es resuelto o se define un workaround aceptado por el equipo.
  • La suite de smoke pasa al 100% antes de retomar la ejecución completa.

9. Gestión de Defectos

9.1 Clasificación de severidad

SeveridadCriterioEjemploBloquea el sprint
CRITICAL El sistema falla completamente, pérdida de datos o vulnerabilidad de seguridad grave Login roto para todos los usuarios; pérdida de datos de empleados Sí – inmediato
HIGH Funcionalidad principal no disponible sin workaround No se pueden crear empleados; widget de clima con spinner infinito Sí
MEDIUM Funcionalidad con workaround disponible o validación incorrecta Contraseña débil aceptada; fecha inválida en Qualifications No
LOW Problema cosmético o de experiencia de usuario menor Botón Reset no limpia todos los filtros No

9.2 Flujo del ciclo de vida de un defecto

Flujo: OPEN (tester reporta) → IN_PROGRESS (developer toma el bug) → RESOLVED (developer cierra con fix) → CLOSED (tester verifica la corrección)

Si la verificación falla → el bug vuelve a OPEN con nota de "reapertura".

9.3 Información requerida por bug

  • Título: breve y descriptivo (qué falla y dónde).
  • Módulo / Sprint: para trazabilidad.
  • Severidad y Prioridad.
  • Pasos para reproducir: paso a paso numerado desde el estado inicial.
  • Resultado esperado: comportamiento correcto según criterio de aceptación.
  • Resultado actual: lo que realmente ocurre.
  • Reportado por / Asignado a.

10. Riesgos y Mitigación

RiesgoProbabilidadImpactoMitigación
El demo de OrangeHRM no está disponible (caída del servidor) Media Alto Reintentar más tarde. Los tests de API del Clima pueden ejecutarse en paralelo. Tener un ambiente local de OrangeHRM como backup (Docker image disponible).
Datos del demo modificados por otros testers simultáneos Alta Medio Tests resilientes: usan datos existentes de la tabla, no dependen de datos fijos. Los datos creados usan timestamp para ser identificables.
API key de WeatherAPI.com expirada o inválida Baja Alto (bloquea Sprint N+3) Renovar la key antes del Sprint N+3. Tener una key de backup. El plan free tiene 1M calls/mes.
Proveedor de WeatherAPI.com cambia el contrato de la API Baja Medio Este es precisamente el riesgo que mitigan las pruebas de contrato (CP-API-001..020). Ante un cambio, los tests fallan y alertan al equipo.
Los testers no pueden completar toda la casuística en el sprint Media Medio Priorizar smoke y regresión. El tester de performance puede apoyar en funcional en sprints con menos carga de performance. Se puede proponer sumar un tester puntual.
Cambios de última hora en el frontend que rompen los selectores de Playwright Media Bajo Usar selectores semánticos (getByRole, getByText) en lugar de CSS frágiles. Reservar día 8 del sprint para estabilizar los tests.

11. Criterios de Suspensión y Reanudación

11.1 Suspensión

  • El ambiente de pruebas no responde por más de 4 horas consecutivas.
  • Un bug CRITICAL bloquea la ejecución de más del 50% de los casos.
  • Más del 30% de los smoke tests fallan por razones no relacionadas con los cambios del sprint (indicador de regresión sistémica).

11.2 Reanudación

  • El ambiente es restaurado y verificado con un smoke básico de infraestructura.
  • El bug CRITICAL es corregido (o se define un workaround aceptado por el PO).
  • La smoke suite pasa al 100% antes de retomar la ejecución completa.
  • El Tester Líder comunica la reanudación al equipo y actualiza el calendario si corresponde.
