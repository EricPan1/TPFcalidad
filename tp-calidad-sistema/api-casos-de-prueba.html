1. Contexto y Objetivo

El Widget de Clima del Sprint N+3 consume la API pública de WeatherAPI.com para mostrar en el dashboard principal información del tiempo en tiempo real. Ante actualizaciones del proveedor, el equipo de QA debe garantizar que la API sigue cumpliendo el contrato que el frontend consume.

Las pruebas de esta sección siguen el enfoque de contract testing: verifican estructura del JSON de respuesta, tipos de datos, códigos HTTP y tiempos de respuesta, independientemente de los valores exactos que retorne (ya que el clima varía).

Tipo de pruebas: Contract Testing (Pruebas de Contrato) Se valida qué forma tiene la respuesta (campos, tipos, códigos HTTP), no qué valores exactos contiene (temperatura, condición, etc.). Esto permite detectar cambios en la API del proveedor antes de que rompan el frontend.

2. Colección Postman

2.1 Ubicación del archivo

La colección está en el repositorio en formato Postman Collection v2.1:

tp-calidad-sistema/ └── api-tests/ └── collections/ ├── WeatherAPI.postman_collection.json ← colección con 20 casos y scripts de test └── WeatherAPI.postman_environment.json ← variables (base_url, api_key, ciudades, fechas)

2.2 Cómo importar en Postman Desktop

  1. Abrir Postman Desktop.
  2. Clic en Import (botón superior izquierdo).
  3. Arrastrar o seleccionar WeatherAPI.postman_collection.json.
  4. Repetir el paso para WeatherAPI.postman_environment.json.
  5. En el selector de entorno (arriba a la derecha), seleccionar "WeatherAPI - Entorno de Pruebas".
  6. En la variable api_key, reemplazar el valor por tu API key real de WeatherAPI.com.
  7. Clic en Run collection para ejecutar todos los casos.

2.3 Cómo ejecutar con Newman (automático, sin instalar nada)

cd tp-calidad-sistema # Ejecuta los 20 casos y genera reporte HTML en api-report/index.html docker compose run --rm api-tests # Ver el reporte en el navegador: # http://localhost:8080/api-report/

4. Realtime Weather API – GET /current.json

Descripción: Retorna el estado del tiempo en tiempo real para una ubicación dada. Es el endpoint principal que consume el widget del dashboard para mostrar la temperatura actual.

ParámetroTipoRequeridoDescripción
keystringSíAPI key de WeatherAPI.com
qstringSíUbicación: ciudad, código postal, coordenadas lat,lon o IP
aqistringNo"yes"/"no" – incluir datos de calidad del aire
langstringNoIdioma de la respuesta (ej: "es" para español)
CP-API-001  ·  Ciudad válida retorna datos de tiempo actual   SMOKE HAPPY   PASÓ
ObjetivoVerificar que una solicitud con ciudad válida y key válida retorna 200 con la estructura de datos completa.
RequestGET /current.json?key={{api_key}}&q=Buenos Aires&aqi=yes
PrecondiciónAPI key activa de WeatherAPI.com configurada en el entorno.
Resultado esperadoHTTP 200 OK. Response contiene objetos location y current con todos los campos requeridos por el widget.
Assertions (test scripts)
  1. Status code es 200
  2. Tiempo de respuesta < 3000ms
  3. location.name es string no vacío
  4. location.country es string
  5. location.lat es number
  6. location.lon es number
  7. location.localtime es string
  8. current.temp_c es number
  9. current.temp_f es number
  10. current.humidity es number
  11. current.wind_kph es number
  12. current.feelslike_c es number
  13. current.condition.text es string no vacío
  14. current.condition.icon es string
  15. Content-Type incluye "application/json"
Resultado obtenidoHTTP 200. Todos los campos presentes con tipos correctos. Tiempo de respuesta promedio: ~350ms.
CP-API-002  ·  Coordenadas GPS retornan datos correctos   HAPPY   PASÓ
ObjetivoVerificar que el endpoint acepta coordenadas geográficas como parámetro q y retorna datos de la ubicación correcta.
RequestGET /current.json?key={{api_key}}&q=-34.6037,-58.3816
PrecondiciónAPI key activa configurada.
Resultado esperadoHTTP 200. La latitud retornada es aproximadamente -34.6 (Buenos Aires).
Assertions
  1. Status code es 200
  2. location.lat existe y Math.abs(lat - (-34.6)) < 1
  3. current.last_updated es string que coincide con patrón de fecha YYYY-MM-DD
Resultado obtenidoHTTP 200. Latitud retornada: -34.61 (corresponde a Buenos Aires). Fecha de actualización correcta.
CP-API-003  ·  Sin API key retorna 401 Unauthorized   UNHAPPY   PASÓ
ObjetivoVerificar que el endpoint rechaza correctamente solicitudes sin el parámetro key.
RequestGET /current.json?q=Buenos Aires (sin key)
PrecondiciónNo enviar el parámetro key.
Resultado esperadoHTTP 401. Body con objeto error que contiene código y mensaje descriptivo.
Assertions
  1. Status code es 401
  2. error.code es number
  3. error.message es string con longitud > 0
Resultado obtenidoHTTP 401. error.code: 1002. error.message: "API key not provided."
CP-API-004  ·  API key inválida retorna 403 Forbidden   UNHAPPY   PASÓ
ObjetivoVerificar que el endpoint rechaza API keys incorrectas con un error de autorización.
RequestGET /current.json?key=INVALID_KEY_123ABC&q=Buenos Aires
Resultado esperadoHTTP 403. error.code en [2006, 2007, 2008].
Assertions
  1. Status code es 403
  2. error.code está en el conjunto [2006, 2007, 2008]
Resultado obtenidoHTTP 403. error.code: 2006 ("API key provided is invalid").
CP-API-005  ·  Ciudad inexistente retorna 400 con error 1006   UNHAPPY   PASÓ
ObjetivoVerificar que el endpoint retorna error controlado cuando la ubicación no existe.
RequestGET /current.json?key={{api_key}}&q=XYZ_CIUDAD_INEXISTENTE_12345
Resultado esperadoHTTP 400. error.code = 1006 ("No matching location found").
Assertions
  1. Status code es 400
  2. error.code es exactamente 1006
Resultado obtenidoHTTP 400. error.code: 1006. error.message: "No matching location found."

5. Forecast Weather API – GET /forecast.json

Descripción: Retorna el pronóstico del tiempo para los próximos N días (máx. 14 en plan gratuito). Cada día incluye datos hora a hora (24 registros), temperatura máxima/mínima y condición.

ParámetroTipoRequeridoDescripción
keystringSíAPI key
qstringSíUbicación
daysintegerNoNúmero de días de pronóstico (1–14). Default: 1
alertsstringNo"yes"/"no" – incluir alertas meteorológicas
aqistringNo"yes"/"no" – incluir calidad del aire
CP-API-006  ·  Pronóstico de 3 días retorna estructura correcta   SMOKE HAPPY   PASÓ
ObjetivoVerificar que el endpoint retorna exactamente 3 días de pronóstico con la estructura completa requerida por el widget.
RequestGET /forecast.json?key={{api_key}}&q=Buenos Aires&days=3&aqi=no&alerts=no
Resultado esperadoHTTP 200. forecast.forecastday es array de 3 elementos. Cada elemento tiene date, day (con maxtemp_c, mintemp_c, condition) y hour (array de 24 elementos).
Assertions
  1. Status code es 200
  2. Response tiene propiedades location, current y forecast
  3. forecast.forecastday.length === 3
  4. Cada día tiene date (string), day (object) y hour (array)
  5. hour.length === 24 en cada día
  6. day.maxtemp_c es number
  7. day.mintemp_c es number
  8. day.condition.text es string
Resultado obtenidoHTTP 200. forecastday contiene 3 días con toda la estructura requerida. 24 registros por hora en cada día.
CP-API-007  ·  Pronóstico de 1 día retorna solo un día   HAPPY   PASÓ
ObjetivoVerificar que con days=1 se retorna exactamente un día y que la fecha es hoy o futura.
RequestGET /forecast.json?key={{api_key}}&q=Buenos Aires&days=1
Assertions
  1. Status code es 200
  2. forecastday.length === 1
  3. La fecha del pronóstico es mayor o igual a la fecha de hoy
Resultado obtenidoHTTP 200. Un único día de pronóstico con fecha igual a hoy.
CP-API-008  ·  Sin parámetro days usa valor por defecto   UNHAPPY   PASÓ
ObjetivoVerificar comportamiento cuando se omite el parámetro days (debe retornar 200 con al menos 1 día).
RequestGET /forecast.json?key={{api_key}}&q=Buenos Aires
Assertions
  1. Status code es 200 (no error)
  2. forecast.forecastday es array con length > 0
Resultado obtenidoHTTP 200. Retorna 1 día por defecto (comportamiento documentado).
CP-API-009  ·  Ciudad inválida retorna 400   UNHAPPY   PASÓ
RequestGET /forecast.json?key={{api_key}}&q=XYZ_CIUDAD_INEXISTENTE_12345&days=3
Assertions
  1. Status code es 400
  2. error.code === 1006
Resultado obtenidoHTTP 400. error.code 1006.
CP-API-010  ·  Sin API key retorna 401   UNHAPPY   PASÓ
RequestGET /forecast.json?q=Buenos Aires&days=3 (sin key)
Assertions
  1. Status code es 401
  2. Response contiene propiedad error
Resultado obtenidoHTTP 401. error.code 1002.

6. History Weather API – GET /history.json

Descripción: Retorna datos históricos del tiempo para una fecha pasada específica. Útil para el widget si se desea mostrar comparación con días anteriores.

ParámetroTipoRequeridoDescripción
keystringSíAPI key
qstringSíUbicación
dtstringSíFecha inicial en formato YYYY-MM-DD (debe ser pasada)
end_dtstringNoFecha final para solicitar rango de fechas (YYYY-MM-DD)
CP-API-011  ·  Fecha histórica válida retorna datos hora a hora   SMOKE HAPPY   PASÓ
ObjetivoVerificar que una fecha pasada válida retorna 24 registros horarios con temperatura en Celsius y Fahrenheit.
RequestGET /history.json?key={{api_key}}&q=Buenos Aires&dt=2024-01-15
Resultado esperadoHTTP 200. forecastday[0].date === "2024-01-15". hour es array de 24 elementos. Cada hora tiene temp_c y temp_f.
Assertions
  1. Status code es 200
  2. Response tiene location y forecast
  3. forecast.forecastday es array
  4. forecastday[0].date === "2024-01-15"
  5. forecastday[0].hour.length === 24
  6. Para las primeras 3 horas: temp_c es number y temp_f es number
Resultado obtenidoHTTP 200. Fecha retornada: 2024-01-15. 24 registros horarios completos con temperatura en ambas unidades.
CP-API-012  ·  Rango de fechas retorna múltiples días   HAPPY   PASÓ
RequestGET /history.json?key={{api_key}}&q=Buenos Aires&dt=2024-01-15&end_dt=2024-01-17
Assertions
  1. Status code es 200
  2. forecastday.length > 1
Resultado obtenidoHTTP 200. forecastday con 3 días (15, 16 y 17 de enero 2024).
CP-API-013  ·  Fecha futura retorna error 400   UNHAPPY   PASÓ
ObjetivoVerificar que el endpoint rechaza fechas futuras ya que no existe historial para ellas.
RequestGET /history.json?key={{api_key}}&q=Buenos Aires&dt=2099-12-31
Assertions
  1. Status code es 400
  2. error.code y error.message presentes
Resultado obtenidoHTTP 400. error.message indica que la fecha está fuera del rango permitido.
CP-API-014  ·  Sin parámetro dt retorna error 400   UNHAPPY   PASÓ
RequestGET /history.json?key={{api_key}}&q=Buenos Aires (sin dt)
Assertions
  1. Status code es 400
  2. error.message no vacío
Resultado obtenidoHTTP 400. Error indicando que el parámetro dt es requerido.
CP-API-015  ·  Sin API key retorna 401   UNHAPPY   PASÓ
RequestGET /history.json?q=Buenos Aires&dt=2024-01-15 (sin key)
Assertions
  1. Status code es 401
Resultado obtenidoHTTP 401.

ParámetroTipoRequeridoDescripción
keystringSíAPI key
qstringSíUbicación
CP-API-016  ·  Solicitud válida retorna estructura de alertas   SMOKE HAPPY   PASÓ
ObjetivoVerificar que el endpoint retorna la estructura correcta independientemente de si hay alertas activas o no.
RequestGET /alerts.json?key={{api_key}}&q=Buenos Aires
Resultado esperadoHTTP 200. Response contiene location y alerts. alerts.alert es siempre un array (puede estar vacío).
Assertions

  2. Response tiene propiedad location
  3. Response tiene propiedad alerts
  4. alerts.alert es un array
  5. Tiempo de respuesta < 3000ms
Resultado obtenidoHTTP 200. Estructura correcta. alerts.alert = [] (sin alertas activas en Buenos Aires al momento de la ejecución).
